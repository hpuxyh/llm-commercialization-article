亏的是豆包，赚的是字节？
大模型商业化真正算的是入口账

有一个很有意思的现象。

如果你同时用过不同大模型产品，可能会发现：AI 已经不只是回答问题了。千问可以接入淘宝、闪购和支付宝；豆包也已经开始支持商品卡、“帮你选”和购物下单。

这类能力短期未必贡献很多收入，但在成熟期会变成一块很大的商业化入口。按本文的情景拆分，只算交易完成后的 Agent 抽佣，约 160 亿，占成熟期 950 亿收入的 17% 左右；如果把商品推荐、AI 答案广告也算进来，广告 + 抽佣合计约 380 亿，占比接近 40%（均为推测的）。

也就是说，问题已经不是“大模型会不会卖货”，而是：为什么它们一定会走到卖货、广告和抽佣这一步？以及不同平台为什么节奏不一样？

表面看，这是产品功能差异。

但如果往深一层看，这其实不是“技术路线不同”，而是商业化账本完全不同。

01 先算 API 的天花板

大模型平台最先能收费的地方，是 API 和 token。这个收入看起来很直接：调用越多，收入越高。

但如果认真算一遍，会发现它很难单独撑起一个超级入口。

我们先拿豆包做样本。这里不是豆包或字节的财报，而是以公开披露的豆包调用量为锚点，做一个量级推算。

豆包大模型公开披露的日均调用量已经到约 120 万亿 token。这个规模很大，但它不能直接按官方 API 标价相乘，因为里面不全是可收费的外部 API。

为了让这笔账更容易看，本文用一个简化情景：假设这 120 万亿 token 里，约 60% 来自 C 端免费使用、内部调用、缓存命中和其他不直接收费的场景；约 40% 来自 API 或企业客户，可以进入结算。

再假设这 40% 可收费 token 的平均结算价约为 0.75 元/百万 token（推测的）。这个价格低于官方标价，是因为大客户折扣、输入输出结构、低价模型和不同调用场景都会拉低平均到手价格。这样摊到全部 token 上，就是 40% × 0.75 元 ≈ 0.3 元/百万 token。

按这个简化口径，豆包模型调用收入大约是 131 亿元级别（推测的）。即使把全量摊薄后的价格提高到 0.8 元，年化也只有约 350 亿元（推测的）。

这已经不是小生意，但问题在于：大模型平台面对的不是百亿级成本，而是千亿级投入。

财务上的第一道坎是：
如果只靠 API 覆盖 1350 亿成本压力，全量摊薄价需要约 3.1 元/百万 token；如果覆盖 2400 亿成熟期成本压力，则需要约 5.5 元/百万 token。它要求 C 端免费、内部调用、企业折扣和低价模型都被充分变现，短期很难成立。

本节口径

样本口径：豆包大模型，不是全行业平均值。公开锚点是豆包大模型日均调用量约 120 万亿 token/天（来源：新华网 / 科技日报引述火山引擎披露）。

价格口径：官方 API 标价不低。以豆包 1.6 的主流 0-32K 输入区间为例，输入 0.8 元/百万 token、输出 8 元/百万 token；按 3:1 输入输出比例折算，综合价约 2.6 元/百万 token。但 120 万亿 token 不是全部都按这个价格对外收费，所以不能直接用官方标价乘总量。

0.3 元怎么理解：本文为了便于读者理解，采用一个简化情景：60% token 不直接收费，40% token 进入 API/企业结算；这 40% 的平均到手价按约 0.75 元/百万 token 估算。于是全量摊薄价 = 40% × 0.75 ≈ 0.3 元/百万 token。这里的 60%、40% 和 0.75 元都不是公司披露数据，而是为了做量级测算的简化假设。

收入计算：120 万亿 ÷ 100 万 = 1.2 亿个“百万 token 单位/天”；1.2 亿 × 0.3 元 ≈ 3600 万元/天；3600 万元 × 365 ≈ 131 亿元/年。若全量摊薄价提高到 0.5 元，年化约 219 亿元；提高到 0.8 元，年化约 350 亿元。

API 覆盖成本所需价格：1350 亿 ÷ 365 ÷ 1.2 亿 ≈ 3.1 元/百万 token；2400 亿 ÷ 365 ÷ 1.2 亿 ≈ 5.5 元/百万 token。这里的 3.1 和 5.5 都是“全量摊薄价”，不是少数外部 API 客户的标价。

02 更关键的问题：成本是另一条完全不同的曲线

如果只看收入，会误判。

因为大模型真正的成本结构，不是“开发完以后边际成本趋近于零”的互联网软件，而是随着使用深度不断放大的计算网络。一次普通问答有推理成本，长上下文会拉高成本，多轮 Agent 调用会继续叠加成本，图像和视频生成又是更重的一层。再往后，还有带宽、存储、训练和持续迭代。

  • 推理成本：每一次回答都要算力
  • 长上下文成本：读得越长，账单越厚
  • 多轮 Agent 成本：搜索、比较、调用工具、确认支付会多次消耗模型
  • 多模态成本：图像、视频、语音比普通文本更重
  • 基础设施成本：带宽、存储、训练和模型迭代持续发生

这意味着：用户用得越多，不一定利润越高，反而可能成本增长越快。尤其是 Agent 场景，用户前台只说一句“帮我买”，后台却可能经历搜索、比价、读评价、调工具、确认支付和生成解释等多轮调用。用户感觉路径变短，平台背后的调用链反而变长。

如果把模型业务做一个整体测算，2026 年收入，包括订阅、API、广告试水和早期 Agent，在保守基准下大约 230 亿（推测的）。同期的成本压力，如果把算力折旧/租赁、训练、推理、研发运营和基础设施摊销一起看，中心情景大约 1350 亿（推测的）。

在这个中心情景下，缺口和覆盖率就不是再推测，而是公式。无论用中心情景，还是放宽到 1100-1600 亿的成本区间，结论都一样：模型收入短期很难覆盖模型成本压力。

03 所以，平台必须做 API 之外的收入

到这里，财务上的必要性就出来了。

平台当然会继续卖 API，也会继续做订阅和企业服务。但如果只靠“每百万 token 收多少钱”，它面对的是一个很硬的约束：要么把全量有效价格大幅提高，要么把成本大幅压下去。前者会受到价格战、免费 C 端和企业折扣限制；后者会发生，但很难无限快地覆盖使用量增长。

所以平台必须找第三条路：不是只提高每个 token 的价格，而是提高每一次用户决策、每一次交易结果的收入密度。

这就是为什么广告和 Agent 抽佣不是“想不想做”的选择，而是财务模型推出来的结果。广告卖的是决策入口里的曝光；Agent 抽佣卖的是交易完成后的结果。它们的共同点，是不再只按调用次数收费，而是按用户决策链路里的价值收费。

更反直觉的是，哪怕这些新收入长出来，缺口也未必立刻缩小。原因是，收入增长和成本增长不是同一条曲线。收入是慢变量，要靠产品、付费、广告和抽佣慢慢成熟；成本是快变量，只要调用量、上下文长度、多模态和 Agent 任务上来，就会立刻放大。

所以即使未来进入成熟阶段，收入可能到 950 亿（推测的），成本压力也可能到 2400 亿（推测的）。覆盖率从约 17% 提升到约 40%（推算结果），看起来更好了；但缺口反而从 1120 亿扩大到 1450 亿（推测的）。

2026 保守基准（推测的）230 亿收入 / 1350 亿成本压力1350 是中心情景；缺口=1350-230=1120，覆盖率≈17%。
成熟期测算（推测的）950 亿收入 / 2400 亿成本压力四条腿合计 950；缺口=2400-950=1450，覆盖率≈40%。

这就是本文最重要的财务判断：
大模型平台不是 API 赚不到钱，而是 API 的收入密度很难独自覆盖入口级产品的成本压力。所以它必须向广告、交易和集团账本寻找第二、第三条腿。

本模块口径

2026 成本压力：1350 亿不是豆包 App 独立财报成本，而是偏“集团 AI 投入口径”的中心情景。外部锚点是公开报道中的字节 2026 年 AI capex 2000 亿元以上；再按豆包、火山方舟、模型训练、推理服务、多模态和 Agent 基础设施占用 55%-60% 估算约 1100-1200 亿，并补入训练实验、模型迭代、数据处理、安全审核、带宽存储、工程研发和运营等 150-250 亿。

2026 公式：缺口 = 1350 - 230 = 1120 亿；覆盖率 = 230 ÷ 1350 ≈ 17%。如果成本压力取 1100 亿，覆盖率约 21%；如果取 1600 亿，覆盖率约 14%。

为什么必须做 API 之外的收入：如果只靠模型调用覆盖 1350 亿成本压力，全量摊薄价要约 3.1 元/百万 token；而本文 2026 保守口径是 0.3 元/百万 token。差距不是单纯提价能解决，因为 120 万亿 token 中包含大量免费 C 端、内部调用、折扣客户和低价模型。

成熟期收入不是公司预测，而是四条腿情景拆分。订阅约 150 亿：以豆包 3 亿级 MAU 为用户池，假设成熟期付费率约 4%-5%、月均客单价约 60-70 元，则年收入约 100-150 亿，取上沿 150 亿。

API/企业调用约 420 亿：2026 保守口径是全量摊薄价 0.3 元/百万 token，对应约 131 亿；成熟期假设可收费 token 占比、企业调用占比和平均结算价上升，把全量摊薄价提高到约 1 元/百万 token，则 1.2 亿个“百万 token 单位/天” × 1 元 × 365 ≈ 438 亿，扣除折扣和低价模型影响后取约 420 亿。

AI 答案广告约 220 亿：不是新增一个凭空市场，而是假设搜索广告、信息流广告、电商广告中有一小部分预算迁移到 AI 答案和推荐位。若可迁移广告预算池按 3000-5000 亿看，迁移 4%-6% 就是约 120-300 亿，中心情景取约 220 亿。

Agent 抽佣约 160 亿：按“被 Agent 影响并可归因成交”的 GMV 来算，而不是按全网 GMV 直接抽成。若成熟期可归因成交额约 2000-4000 亿，平台净佣金/服务费率约 4%-8%，对应约 80-320 亿，中心情景取约 160 亿。只算交易抽佣，160 ÷ 950 ≈ 17%；若把 AI 答案广告也算作“商品推荐/交易入口收入”，220 + 160 = 380 亿，占成熟期收入约 40%。四条腿合计 150 + 420 + 220 + 160 = 950 亿；成本压力按 2400 亿情景，则缺口 = 2400 - 950 = 1450 亿，覆盖率约 40%。

04 真正的核心问题：入口怎么赚钱

有了上面的账，问题可以收敛成一句话：大模型平台的问题，从来不是“能不能卖 API”，而是“入口怎么赚钱”。

但这件事不能只从平台视角理解。广告主和平台之所以会沿着这条路走，是因为用户习惯正在变：用户先从“自己搜索十个链接”，变成“直接问 AI 要一个答案”；再从“自己拿答案去操作”，变成“让 Agent 直接完成任务”。

用户每少做一步，商业入口就往下游挪一步。用户不再点链接，品牌就会关心自己有没有出现在 AI 答案里；用户不再自己下单，商家就会关心自己能不能被 Agent 选中。平台能收费的位置，也就从“模型调用”移动到“决策露出”，最后移动到“交易结果”。

所以 API 只是第一阶段。真正的路径其实是三步：先卖能力，再卖曝光，最后卖结果。

  • 第一阶段：卖能力。用户把 AI 当工具用，平台卖 API、订阅、企业服务。
  • 第二阶段：卖曝光。用户把 AI 答案当决策入口，广告主为推荐、品牌露出和“被提及”付费。
  • 第三阶段：卖结果。用户把执行交给 Agent，商家为下单、履约和交易完成后的结果付费。

05 为什么广告是必然的，而不是“想不想做”

很多人会误解一个问题：大模型为什么一定要做广告？

答案其实不是战略选择，而是结构迁移。

过去互联网是这样的：用户搜索，点击链接，比较信息，再做决策。广告挂在搜索结果、信息流、电商列表里。

但 AI 出现后，用户越来越少看十个链接，而是直接看一个答案。

这带来一个结构性变化：入口从“链接列表”变成“单一答案”。

于是广告不会消失，只会迁移：从搜索结果迁移到 AI 答案，从信息流迁移到推荐结果，从点击迁移到被“提及”。

一句话总结：
钱不会消失，只是换了入口。

06 为什么 Agent 抽佣是第二条必然路径

如果说广告是“迁移”，那 Agent 是“升级”。

关键变化是，过去用户自己做所有决策：比价、选商品、看评价、填信息、下单。未来则变成：用户只说目标，Agent 负责执行。

比如，“帮我点个附近评分最高的外卖”；“帮我订一张性价比最高的机票”；“帮我选一个适合送礼的商品”。

只要 Agent 做到三件事：更省时间、不更贵、结果可靠，用户就会接受它替代自己决策。

而一旦这件事发生，商业模式自然会从“信息收费”变成“交易抽佣”。

本质变化是：
从“帮你找信息”，变成“帮你完成结果”。

本模块口径

三阶段不是严格时间表，而是收入重心迁移：先卖调用能力，再卖决策入口里的曝光，最后在交易完成后收费。推动它变化的不是平台单方面想收费，而是用户体验变化：用户少点一次链接，品牌就少一次传统曝光；用户少做一次下单动作，商家就需要争取被 Agent 选中。

图里的 220 亿 AI 答案广告、160 亿 Agent 抽佣，是成熟期收入拆分里的情景数，不是公司披露收入。220 亿对应搜索广告、信息流广告和电商广告中一部分预算迁移到 AI 答案和推荐位；160 亿对应 Agent 承担比价、推荐、下单和履约入口后，平台从部分交易结果里收取佣金。

用户不会因为“平台要抽佣”而选择 Agent。用户选择 Agent 的理由是更省时间、结果可靠、价格不吃亏；佣金通常不是直接向用户收，而是藏在商家获客、履约或交易分成里。

07 三个玩家，三种完全不同的路径

① 千问（阿里）

千问背后是淘宝、支付宝、本地生活和履约体系。它的优势是有完整交易闭环。

所以 Agent 对它来说，不是破坏体系，而是增强体系。它可以更激进地做“直接下单”。

② 豆包（字节）

豆包背后是内容流、广告系统、种草链路和电商转化。更准确地说，豆包不是没有做交易入口：它已经开始用商品卡、“帮你选”和购物下单，把 AI 对话接到抖音电商。

但这恰恰说明问题已经进入下一层：当用户在豆包里直接看商品、比价、下单，原来“内容种草 → 广告投流 → 抖音商城转化”的链路就会被压缩。

所以豆包的难点不是“能不能做”，而是“怎么做得足够快，又不让旧的广告和内容电商机器被自己过早蚕食”。

③ DeepSeek

DeepSeek 没有同等级的交易系统，也没有同等级的广告体系。它更可能走低价、中立、基础模型路线。

它的优势反而是：不被商业化路径绑架。

玩家判断口径

千问接入淘宝、闪购、支付宝属于公开事实；“更适合激进做交易型 Agent”是基于阿里交易闭环的战略判断。

豆包支持商品卡、购物下单和“帮你选”来自媒体实测与报道；“需要权衡旧广告/内容电商链路”是从字节现有内容电商、信息流广告和种草投流链路可能被 Agent 缩短来推断，不是字节官方表态。

DeepSeek 的“低价、中立、基础模型”判断，来自它缺少同等级广告和交易闭环这一结构条件。

08 真正的本质：不是模型竞争，是入口竞争

如果把所有东西压缩成一句话：大模型不是在卖能力，而是在争夺下一代入口。

入口一旦变了，搜索广告会变，内容分发会变，电商转化链路会变，甚至“决策权”本身都会被重新定价。

09 最后一个更底层的结论

回到最开始的问题：为什么看起来豆包在亏钱，但字节还在继续做？

因为你看到的是“产品账”，但它算的是“集团账”。

豆包本身可能是高成本、低直接收入、强补贴。但在集团层面，它的价值是下一代入口、广告理解能力升级、电商转化效率提升，以及长期 AI 操作系统入口。

所以“亏的是豆包，赚的是字节”，更准确地说，不是豆包亏损不重要，而是豆包不能只按一个独立 App 的短期损益来理解。它更像字节押注下一代入口的一张期权。

本模块口径

“搜索框 → 信息流 → AI 对话 → Agent 执行”不是说旧入口会立刻消失，而是说新增入口会逐步拿走一部分决策时间。商业化跟着入口走：用户在哪里做决定，广告、推荐、佣金和商家预算就会往哪里迁移。

这里不是说豆包亏损不重要，而是说豆包不能只按独立 App 的短期损益判断。它带来的收益可能不记在豆包账上，而是体现在广告提效、电商转化、火山引擎企业服务，以及防御下一代入口被别人拿走。

结语

大模型商业化的本质不是“模型怎么收费”，而是一个更大的问题：谁能定义下一代互联网入口。

当入口从搜索框变成 AI 对话框，再变成 Agent 执行链路之后，广告会迁移，交易会迁移，决策也会迁移。

最终留下来的问题只有一个：谁能在不破坏旧体系的情况下，接住新入口。

数据口径说明

1. 有公开论据的事实锚点：新华网 / 科技日报关于火山引擎披露豆包大模型 120 万亿 Token 的报道用于核对 token 规模；新华网关于豆包 1.6 输入 0.8 元/百万 token、输出 8 元/百万 token 的报道用于核对 API 标价；火山方舟模型价格页用于核对官方价格口径；SCMP 关于字节 2026 AI capex 超 2000 亿元的报道用于核对成本量级背景；QuestMobile 2026Q1 AI 应用报告用于核对豆包、千问、DeepSeek 月活；科技日报千问接入淘宝、闪购、支付宝报道用于核对交易型 Agent 路径；新浪科技关于豆包支持应用内购物下单的实测报道与 i黑马关于“豆包帮你选”上线的报道用于核对豆包购物入口；支付宝 AI 付官网用于核对智能体支付能力；OpenAI Instant Checkout / Agentic Commerce Protocol与 Stripe 官方新闻稿用于核对 Agent 交易协议趋势；DeepSeek API Pricing用于核对模型价格口径。

2. 公式推算 / 推测数据：0.3 元/百万 token 是一个简化测算口径，不是 API 标价；约 131 亿模型调用收入、230 亿总收入、1350 亿成本压力、950 亿成熟收入、2400 亿成熟期成本压力、1120 亿 / 1450 亿缺口，均属于量级测算或情景推演，不是公司财报。核心公式是：模型调用收入 = token 调用量 ÷ 100 万 × 全量摊薄价 × 天数；全量摊薄价 = 可收费 token 占比 × 可收费 token 平均结算价。本文保守情景取 40% × 0.75 元 ≈ 0.3 元/百万 token；API 覆盖成本所需摊薄价 = 成本压力 ÷ 365 ÷ 日 token 单位；成本压力 = AI 基础设施资源分摊 + 训练/推理 + 研发运营/带宽存储；覆盖率 = 收入 ÷ 成本压力；缺口 = 成本压力 - 收入。950 亿成熟收入的拆分公式是：订阅收入 = MAU × 付费率 × 月均客单价 × 12；API/企业调用收入 = 日 token 单位 × 成熟期全量摊薄价 × 365；AI 答案广告 = 可迁移广告预算 × 迁移比例；Agent 抽佣 = 可归因成交 GMV × 平台净佣金率。特别注意：120 万亿 token 是豆包大模型公开使用量锚点，不等于“全部都是可收费外部 API 调用”；1350 亿是中心情景，不是公司披露成本。
